Project Summary

Confirmed: - AutoDocX is a FastAPI-based application that scans local Python projects, analyzes their structure, generates an LLM-based "understanding" of the codebase, and produces standalone HTML documentation. The entry point is app/main.py and main API surface is defined in app/api/project_routes.py. The project defines Pydantic models in app/models/project.py and multiple services under app/services/ for scanning, analysis, LLM integration, comparison, and storage. A Streamlit UI helper exists at ui/streamlit_app.py. Reasonable inferences: - The intended user workflow is: point AutoDocX at a local project path, run a scan, optionally request an LLM-generated project understanding, and persist generated HTML documentation; there is an all-in-one sync endpoint to perform the whole workflow in one call. Stored artifacts (scans, understandings, documents) appear under a workspace directory configured by app.core.config.

Architecture Overview

Confirmed: - Components: - API layer: app/api/project_routes.py exposing endpoints under /api/projects. - Core/config: app/core/config.py for environment loading and directory setup. - Services: app/services/* implementing scanners, analyzers, LLM providers, storage, document builder, and comparators. - Models: app/models/project.py providing Pydantic request/response schemas. - UI helpers: ui/streamlit_app.py contains convenience functions for a Streamlit front-end. - Storage modules: ScanStorage, UnderstandingStorage, DocumentStorage persist JSON metadata and HTML files to the workspace (paths referenced in config: UNDERSTANDINGS_PATH, DOCUMENTS_PATH). - LLM integration: Abstracted by BaseLLMProvider with concrete OpenAIProvider and AnthropicProvider implementations. LLM selection is done through a factory LLMProviderFactory. Reasonable inferences: - The system is modular: scanning/parsing (PythonCodeParser + ProjectScanner) -> analysis (ProjectAnalyzer) -> context building (ProjectContextBuilder) -> LLM understanding (ProjectUnderstandingService via LLM providers) -> document generation (DocumentBuilder) -> persistence (ScanStorage/UnderstandingStorage/DocumentStorage). The API orchestrates these steps in endpoints such as /documentation/sync and /{project_name}/documents/update.

Execution Flow

1

Trigger a project scan

API receives a scan request (POST /api/projects/scan or part of the documentation sync). ProjectScanner.scan_project inspects the project path, enumerates files, computes hashes, parses Python files using PythonCodeParser, and calls ProjectAnalyzer to build project-level relationships and statistics.

Persist scan results

The API uses ScanStorage to save the scan JSON under the workspace/scans/<project_name>/<scan_id>.json pattern and returns scan metadata.

3

Build LLM-ready project context

ProjectContextBuilder.build_context converts the stored scan into either a 'detailed' or 'llm' mode context. This context is used as input to LLM prompts.

app.services.project_context_builderapp.services.project_scannerapp.services.project_analyzer
4

Generate structured understanding with an LLM

ProjectUnderstandingService.generate_understanding constructs a system prompt (PROJECT_UNDERSTANDING_SYSTEM_PROMPT) and a user prompt (via build_project_understanding_prompt), creates an LLM provider via LLMProviderFactory (OpenAIProvider or AnthropicProvider), and calls provider.generate_json to obtain a structured JSON understanding.

5

Persist understanding

UnderstandingStorage.save_understanding saves the LLM-produced understanding JSON to workspace/understandings/<project_name>/<understanding_id>.json.

app.services.understanding_storageapp.api.project_routes
6

Generate HTML documentation

DocumentBuilder.build_html builds a standalone HTML string from a stored understanding (escaping content via html.escape). DocumentStorage.save_document persists the HTML file under DOCUMENTS_PATH and writes associated metadata. get_project_document_html returns the file via FastAPI FileResponse.

app.services.document_builderapp.services.document_storageapp.api.project_routes
7

Compare scans and update documents (optional)

When a previous document exists, ScanComparator.compare_scans compares old and new scans to detect file/symbol/route/dependency changes. If documentation-relevant changes exist (comparison_contains_changes), a new understanding/document is generated and saved; otherwise, the existing document is returned.
