Project Summary

CONFIRMED: AutoDocX is a FastAPI-based service that scans a local Python project, analyzes its structure, generates a compact LLM-ready project context, asks a configured LLM provider (OpenAI or Anthropic) to create a structured project understanding, builds standalone HTML documentation from that understanding, and stores scans, understandings, and generated documents. The application exposes REST endpoints under /api/projects to perform scanning, comparison, context-building, understanding generation, document generation/update, and retrieval. KEY ENTRY: app/main.py registers routes and exposes health endpoints. CONFIDENTIAL: No secrets are included in the supplied context. INFERRED (reasonable): The primary user workflow is automated via a single endpoint POST /api/projects/documentation/sync which orchestrates scanning, understanding generation, optional comparison with a prior scan, and document creation. The included UI (ui/streamlit_app.py) is a lightweight client that calls these endpoints to drive synchronization from a GUI. Some CLI/start scripts exist (install.bat, start.bat) for local deployment/testing.

Architecture Overview

CONFIRMED: The system is structured as a service-oriented Python application with layers: API (app.api.project_routes), services (app.services.*), storage adapters (ScanStorage, UnderstandingStorage, DocumentStorage), LLM provider adapters (app.services.llm.*), models/schemas (app.models.project), configuration (app.core.config), and a small UI client (ui/streamlit_app.py). ProjectScanner uses PythonCodeParser and ProjectAnalyzer to produce a stored scan. ProjectContextBuilder transforms a stored scan into either detailed or LLM-focused context. ProjectUnderstandingService uses LLMProviderFactory plus prompt-building utilities to generate a structured JSON understanding. DocumentBuilder converts stored understanding into HTML. Storage classes store and list artifacts under workspace paths and DOCUMENTS/UNDERSTANDINGS paths configured in app.core.config. INFERRED: Communication is synchronous HTTP between client/UI and the FastAPI server. LLM calls are synchronous blocking calls made by provider adapters (OpenAIProvider, AnthropicProvider). The design intentionally separates storage (persistence file layout) from generation logic so documents/understandings/scans are retrievable independently.

ProjectScanner.scan_project traverses the target project path, enumerates files, computes sizes and hashes, and delegates to PythonCodeParser.parse_file for Python files. 4

Build project context

ProjectContextBuilder.build_context converts a stored_scan into either 'detailed' or 'llm' context. This context is used by ProjectUnderstandingService to generate LLM prompts or returned directly via the context endpoint.

app.services.project_context_builder
5

Generate project understanding with LLM

ProjectUnderstandingService.generate_understanding uses LLMProviderFactory.create_provider to instantiate an LLM provider (OpenAIProvider or AnthropicProvider) using API key and model from app.core.config. It builds prompts via build_project_understanding_prompt and sends them to the provider's generate_json which returns parsed JSON understanding. The returned understanding is then saved via UnderstandingStorage.save_understanding.

6

Generate HTML documentation

DocumentBuilder.build_html takes a stored_understanding dict and produces an HTML string (escaping content as needed). DocumentStorage.save_document writes the HTML file and metadata (get_document_file, get_document_metadata, list_documents support retrieval). The API may return FileResponse for direct downloads via get_project_document_html.

Compare scans and optionally update document

If a prior scan/document exists, ScanComparator.compare_scans compares two stored scans producing a diff (added/removed/modified files, symbols, routes, internal dependencies). app.api.project_routes contains logic to determine whether a comparison contains documentation-relevant changes (comparison_contains_changes) and to either return unchanged document metadata or create a new understanding/document when changes are present.

Return result to caller

API endpoints return Pydantic models defined in app.models.project (e.g., ProjectScanResponse, ProjectUnderstandingResponse, DocumentationSyncResponse) or FileResponse for HTML retrieval. The UI client (ui/streamlit_app.py) calls documentation sync and displays identifiers and metadata.

Risks and Gaps

high

Synchronous LLM and I/O calls may block server

LLM provider adapters (OpenAIProvider, AnthropicProvider) expose synchronous generate_json methods and the project overall contains no async endpoints or async I/O patterns in the provided context. Long-running LLM requests could block worker threads/processes and degrade throughput.

high

No authentication/authorization in API surface

The API routes do not indicate any authentication or authorization checks. Exposed endpoints can trigger LLM calls and write to workspace storage. If deployed without network protection, this could be abused or leak project contents.

medium

LLM provider configuration and secrets handling are unclear

app.core.config exposes get_llm_api_key and get_llm_provider but the context does not show secure secret management (vaults, encryption) or rotation. The UI and API likely need clear guidance on how API keys are provided and protected.

medium

Limited provider implementations

Factory contains OpenAIProvider and AnthropicProvider only. Adding new providers requires implementing BaseLLMProvider. Tests or fallbacks for unsupported provider strings are not visible in the context.

medium

Storage concurrency and atomicity not specified

Storage classes appear to write JSON files and HTML to filesystem paths. There is no visible locking, transactional guarantees, or conflict resolution strategy when concurrent requests target the same project (race conditions possible).

low

No explicit input validation samples beyond Pydantic models

Pydantic models exist for most request/response shapes, but the context does not show example values, exhaustive schema constraints, or error mapping for invalid input outside the internal helper _value_error_status mapping.

medium

Unclear logging, monitoring, and error classification

No logging or metrics exports are visible in the provided modules. Observability around LLM failures, timeouts, and storage errors is therefore uncertain from the context.

low

Potential size/costs of LLM prompts and responses

ProjectContextBuilder supports 'detailed' mode that keeps full module content-hashes and possibly larger context payloads. Large prompts could increase latency and LLM cost; no chunking/streaming logic is visible.

low

UI is minimal and tightly coupled to API surface

ui/streamlit_app.py provides helper functions for the FastAPI endpoints but no authentication, retry, or progress UI for lengthy LLM calls. It may not handle partial failures or retries gracefully.
