Architecture Overview

Confirmed: The application is organized as a FastAPI service (app/main.py) which mounts an APIRouter defined in app.api.project_routes. The router delegates work to service classes in app.services: ProjectScanner (scans filesystem and delegates file parsing to PythonCodeParser and analysis to ProjectAnalyzer), ScanStorage (persists and retrieves stored scan JSON), ScanComparator (compares two stored scans), ProjectContextBuilder (converts stored scan into either detailed or LLM mode context), and ProjectUnderstandingService (builds LLM-ready context and uses an LLM provider to generate structured JSON). LLM integrations follow a provider pattern: app.services.llm.base defines BaseLLMProvider; OpenAIProvider and AnthropicProvider implement it; LLMProviderFactory selects and instantiates providers. app.models.project contains Pydantic models used as request/response schemas. app.core.config provides configuration helpers including get_llm_provider, get_llm_api_key, and get_llm_model. Reasonable inferences: Typical request flow is: API handler -> relevant service(s) -> storage/LLM provider -> response. All API handlers are synchronous and likely run inside the FastAPI worker process. Stored scans are placed under a workspace directory (ScanStorage docstring). The prompt builder (app.services.llm.prompts) assembles user/system prompts used by providers. The project is intended to run locally and operate on local filesystem paths supplied to the scan endpoint. Some runtime details (how environment variables are loaded, any background tasks, or rate limiting) are not visible and therefore uncertain.
