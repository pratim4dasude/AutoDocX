Project Summary

Confirmed: AutoDocX is a FastAPI application that scans local Python projects, analyzes source structure, generates LLM-based structured project understandings, compares scans, and builds/stores standalone HTML documentation. The application exposes REST endpoints (under /api/projects/*) to trigger scans, comparisons, context building, understanding generation, and document generation/storage. It includes pluggable LLM providers (OpenAI and Anthropic) implemented behind a BaseLLMProvider interface and created via LLMProviderFactory. Storage components persist scans, understandings, and generated HTML documents to a workspace (paths referenced by constants such as DOCUMENTS_PATH and UNDERSTANDINGS_PATH). Confirmed entry point: app/main.py. Reasonable inferences: The system is intended for automating documentation generation for Python projects by combining static analysis (code parsing + project analysis) with LLM-generated natural-language structure and HTML rendering. The DocumentationSync endpoint implements a single-workflow convenience path (scan -> optionally compare -> generate understanding -> generate & save document) according to the endpoint docstring.

Architecture Overview

Confirmed: The codebase is organized into an API layer (app.api.project_routes), configuration (app.core.config), data models (app.models.project using Pydantic), service modules for scanning/parsing/analyzing (app.services.code_parser, app.services.project_scanner, app.services.project_analyzer), context and understanding generation (app.services.project_context_builder, app.services.project_understanding_service), scan comparison (app.services.scan_comparator), document building and storage (app.services.document_builder, app.services.document_storage), and persistent storage of scans/understandings (app.services.scan_storage, app.services.understanding_storage). LLM integrations live under app.services.llm with provider implementations app.services.llm.openai_provider and app.services.llm.anthropic_provider and a factory app.services.llm.factory. The API uses FastAPI and returns Pydantic models defined in app.models.project. Confirmed that environment/config loading is performed in app.core.config (load_dotenv usage) and that constants like DOCUMENTS_PATH and UNDERSTANDINGS_PATH are referenced. Reasonable inferences: The system is synchronous (no async functions detected) and uses file-based JSON storage for scans and understandings (docstrings state workspace/locations). The LLM provider abstraction suggests the system constructs system/user prompts, sends them to a provider via generate_json, and expects JSON-parsable responses. Exact storage layout details beyond described file paths, concurrency model, and authentication/authorization are not present in the provided context.

Receive API request

A client calls one of the FastAPI endpoints (for example: POST /api/projects/documentation/sync or POST /api/projects/scan). Requests are validated against Pydantic models in app.models.project. The app entrypoint is app/main.py which mounts the router defined in app.api.project_routes.

Project scanning & parsing

Persist scan result

The scan result is saved via ScanStorage.save_scan into the workspace storage path (the docstring indicates files are stored under workspace/scans/<project_name>/<scan_id>.json). The API then can return scan metadata or contents to the caller.

4

Compare scans (optional)

If an earlier scan or document exists and an update flow runs (e.g., DocumentationSync or documents/update), the service retrieves previous scan(s) via ScanStorage.get_latest_scan/get_scan and runs ScanComparator.compare_scans to detect added/removed/modified files, symbols, routes and dependencies. Comparison results are used to decide whether documentation should be updated.

app.services.scan_storageapp.services.scan_comparatorapp.services.document_storage
5

Build LLM-ready context and request understanding

When an understanding is required, ProjectContextBuilder.build_context converts a stored_scan into either detailed or LLM-optimized project context. ProjectUnderstandingService.generate_understanding uses ProjectContextBuilder and build_project_understanding_prompt to construct prompts. The LLM provider is created via LLMProviderFactory.create_provider (choosing OpenAIProvider or AnthropicProvider) and generate_json is called. The provider's response is parsed (BaseLLMProvider.parse_json_response) and the resulting structured understanding is saved via UnderstandingStorage.save_understanding.

6

Generate HTML documentation and save

DocumentBuilder.build_html turns a stored_understanding into standalone HTML (escaping content as necessary). DocumentStorage.save_document saves the HTML file and metadata (DocumentStorage references DOCUMENTS_PATH). The API can return document metadata or serve the HTML file using FastAPI FileResponse.
