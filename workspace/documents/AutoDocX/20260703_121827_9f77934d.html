Project Summary

Confirmed: - AutoDocX is a Python application that scans local Python projects, analyzes their code structure, builds an LLM-ready project context, asks a configured LLM to produce a structured project understanding, and generates/saves standalone HTML documentation. This is implemented as a FastAPI service (entrypoint app/main.py) with a Streamlit UI client (ui/streamlit_app.py). Reasonable inferences: - The primary user workflow is to call /api/projects/documentation/sync (sync_project_documentation) to run an end-to-end flow that scans, compares, understands, and produces or updates documentation. The components persist intermediate artifacts (scans, understandings, documents) under a workspace directory (ScanStorage, UnderstandingStorage, DocumentStorage). The LLM provider is chosen via configuration functions in app.core.config and an LLM factory (app.services.llm.factory) creates the provider (OpenAIProvider or AnthropicProvider). Some filesystem path constants (e.g., DOCUMENTS_PATH, UNDERSTANDINGS_PATH) are defined in app.core.config. Uncertainties: - The exact JSON contract produced by the LLM and the prompt text content are not shown here (prompts module reference exists but full constants are not listed). The concrete storage layout, permissions, and concurrency behavior are inferred from docstrings but not visible in file contents provided.

Architecture Overview

Confirmed: - Layers: - API layer: app.api.project_routes defines HTTP endpoints and orchestrates the workflow using service classes. - Services layer: scanning (ProjectScanner), parsing (PythonCodeParser), analysis (ProjectAnalyzer), context construction (ProjectContextBuilder), LLM orchestration (ProjectUnderstandingService + LLM provider implementations), comparison (ScanComparator), and content generation/storage (DocumentBuilder, DocumentStorage, ScanStorage, UnderstandingStorage). - Configuration layer: app.core.config provides environment-driven configuration (LLM provider, API key, model selection) and directory setup. - UI: ui.streamlit_app contains a Streamlit-based client that calls the FastAPI endpoints. Reasonable inferences: - Data persistence is file-based under a 'workspace' root (ScanStorage.save_scan docstring referencing workspace/scans/<project_name>/<scan_id>.json and similar docstrings in UnderstandingStorage). DocumentStorage references DOCUMENTS_PATH. These imply a filesystem-backed storage strategy rather than a database. - LLM interactions are synchronous blocking calls (provider.generate_json methods are non-async) and responses are parsed to JSON with BaseLLMProvider.parse_json. The factory normalizes provider names and instantiates provider classes. Uncertainties: - Deployment topology (single process, multiple workers) and authentication for the API are not present in the provided context.

Risks and Gaps

high

LLM response parsing fragility

Parsing free-text LLM responses into JSON is error-prone. While BaseLLMProvider.parse_json_response handles several wrapping formats, malformed responses or partial JSON could break generate_understanding or cause stored understandings to be invalid.

high

No authentication/authorization in API surface

There is no evidence of authentication middleware, API keys, or access control on endpoints. Serving stored HTML and providing LLM API keys via configuration may expose sensitive data if deployed without protective measures.

medium

Concurrent access and race conditions

Storage is file-based (workspace/...). Concurrent requests creating scans, understandings, or documents for the same project could race when listing/getting latest artifacts or when saving new files. No locking or transactional behavior is visible from provided modules.

medium

Error handling and HTTP status mapping

Some internal helpers exist (e.g., _value_error_status) to map storage lookup errors to HTTP 404, but full error propagation, validation of LLM responses, and recovery for partial failures are not fully visible.

medium

Secrets management

LLM API keys are obtained via app.core.config.get_llm_api_key, likely from environment variables. There's no shown secret rotation or secure storage; keys may be loaded in plaintext into process env.

low

Testing and observability gaps

No unit tests or logging/metrics modules are visible in the provided context. Observability for LLM calls, long-running scans, and disk IO is not shown.

medium

Unclear prompt and JSON schema

Full prompt text (PROJECT_UNDERSTANDING_SYSTEM_PROMPT) and the exact JSON schema expected from the LLM are not included, which makes it harder to validate or extend the LLM output and to write deterministic tests.
