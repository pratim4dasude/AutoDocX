Project Summary

Confirmed findings: AutoDocX is a Python FastAPI application that scans a local Python project, analyzes code structure, generates an LLM-based structured project understanding, builds standalone HTML documentation from that understanding, and stores scans / understandings / generated documents on disk. The application exposes REST endpoints under the /api/projects prefix (implemented in app.api.project_routes) and a small Streamlit UI (ui.streamlit_app) that calls those API endpoints. Core services include: ProjectScanner (scans filesystem, delegates parsing and analysis), PythonCodeParser (parses Python source files into structure and content hashes), ProjectAnalyzer (builds project-level relationships and endpoint paths), ProjectContextBuilder (produces either detailed or LLM-ready context), ProjectUnderstandingService (invokes an LLM provider via LLMProviderFactory to produce JSON understanding), DocumentBuilder (renders HTML from understanding), and storage layers ScanStorage, UnderstandingStorage, and DocumentStorage which persist JSON metadata and files in a workspace directory and configured DOCUMENTS_PATH/UNDERSTANDINGS_PATH. Reasonable inferences: The primary user workflow is to call the documentation synchronization endpoint (/api/projects/documentation/sync) which orchestrates scanning, optional comparison with prior scans, LLM-based understanding generation, and document creation or update. The system supports multiple LLM providers (OpenAI and Anthropic are implemented) selected at runtime. Configuration (LLM provider, API key, model) appears to be read from environment/config helpers in app.core.config. Persistent storage is file-based JSON under the workspace and paths implied by DOCUMENTS_PATH and UNDERSTANDINGS_PATH; no database dependency is visible. Some UI conveniences (building URLs, showing history) are implemented in the Streamlit app.

Architecture Overview

Confirmed findings: The project is modular with the following layers: (1) HTTP API layer: app.main registers routes provided by app.api.project_routes (APIRouter). Endpoints are synchronous FastAPI handlers. (2) Core services: scanning and parsing (app.services.project_scanner, app.services.code_parser), analysis (app.services.project_analyzer), context building (app.services.project_context_builder), LLM orchestration (app.services.project_understanding_service plus app.services.llm.*), comparison (app.services.scan_comparator), document generation and storage (app.services.document_builder, app.services.document_storage) and persistent scan/understanding storage (app.services.scan_storage, app.services.understanding_storage). (3) Models: pydantic models in app.models.project define request/response schemas. (4) UI: ui.streamlit_app is a client that consumes the API. Reasonable inferences: Data moves from raw filesystem (scanned by ProjectScanner) -> parsed file structures (PythonCodeParser) -> project analysis (ProjectAnalyzer) -> stored scan JSON (ScanStorage) -> LLM-ready context (ProjectContextBuilder) -> LLM provider call (ProjectUnderstandingService using LLMProviderFactory) -> stored understanding JSON (UnderstandingStorage) -> HTML generation (DocumentBuilder) -> saved document and metadata (DocumentStorage). The router app.api.project_routes orchestrates service instantiation and high-level request flows. LLM provider selection and credentials are provided via app.core.config helpers at runtime. The file-based storage and lack of database code suggest the app is intended for local/low-scale usage or as a developer tool.

Risks and Gaps

high

File-based storage scalability and concurrency

ScanStorage, UnderstandingStorage, and DocumentStorage persist JSON and files on the filesystem (workspace and paths). This design may cause concurrency issues, file corruption, and performance bottlenecks under concurrent requests or large projects. There is no evidence of locking, transactional writes, or a database backend.

high

LLM call reliability and error handling

LLM interactions are synchronous and rely on provider SDKs (OpenAI/Anthropic). The provided context does not show retry/backoff logic, timeout handling, or rate-limit handling. Parsing model responses into JSON can fail; while BaseLLMProvider.parse_json_response exists, the robustness of error handling and schema validation for model outputs is not visible.

medium

Secrets and configuration handling

app.core.config exposes get_llm_api_key and get_llm_provider but the capture shows dotenv only. There is limited visibility into secure storage, rotation, or per-request overrides for API keys. The Streamlit UI and API may require API keys in environment variables—details are not shown.

high

Authentication and access control

No authentication, authorization, or access control mechanisms are visible in the API layer. Endpoints that create scans, trigger LLM calls, and download HTML are publicly exposed by default, which may be undesirable for sensitive projects.

medium

Limited observability and monitoring

There is no visible logging configuration, metrics, or tracing in the analyzed modules. Diagnosing failures or LLM issues in production may be difficult without structured logs or metrics.

medium

Missing tests and CI evidence

Scanned files and metadata do not indicate unit tests or integration tests for key flows (LLM provider interactions, parsing, storage). This gap increases risk of regressions when refactoring or introducing new providers.

high

Unclear prompt schema and validation for LLM outputs

While prompts are built by app.services.llm.prompts and responses are parsed into JSON, the exact expected JSON schema of the LLM understanding is not visible. There is uncertainty about validation of the LLM-generated structure before it is saved or used to build documents.

medium

Sync endpoint complexity and single-shot orchestration

The documentation synchronization endpoint orchestrates scanning, comparing, LLM generation, and document building in one request. This long-running synchronous flow may time out or fail if LLM calls are slow. There is no visible asynchronous task queue or background job mechanism.
