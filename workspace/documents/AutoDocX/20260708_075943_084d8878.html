Overview

This project provides a developer-documentation engine (AutoDocX) that scans a local Python codebase, extracts structural metadata, generates an LLM-assisted narrative understanding, and produces standalone HTML developer documentation. It exposes a FastAPI HTTP API for scanning projects, generating and listing scans, producing LLM understandings, comparing scans, and creating or updating HTML documentation; a Streamlit UI client is included to call those APIs.
API paths, handlers, signatures, parameters, classes, and methods are taken from parser data where available, not guessed by the LLM.

Start the FastAPI server, then open the generated API docs or call the project documentation sync endpoint.

Architecture

The application is organized into clear layers: a FastAPI entry layer (app/main.py and app.api.project_routes) that exposes HTTP endpoints; service layers under app.services that implement scanning, parsing, analysis, LLM integration, comparison, runtime context analysis, and document building; storage layers (app.services.scan_storage, app.services.understanding_storage, app.services.document_storage) that persist scans, understanding JSON, and generated HTML plus runtime assets; and a small UI client (ui/streamlit_app). Configuration and workspace paths are centralized in app.core.config. Runtime moves from request -> ProjectScanner (uses PythonCodeParser and ProjectAnalyzer) to produce a scan -> ScanStorage saves the scan -> ProjectUnderstandingService (uses ProjectContextBuilder and LLM provider factory) generates narrative understanding -> UnderstandingStorage saves it -> DocumentBuilder renders HTML -> DocumentStorage saves HTML and any runtime assets. Additional flows support scan comparison, LLM runtime context analysis (vision-capable providers), and recovery of embedded assets from existing HTML files.

Runtime and Tooling Context

Runtime summary

This repository is a Python-based documentation generation service with a web API and a separate Streamlit UI. The backend exposes a FastAPI router (app.api.project_routes) implementing endpoints to scan local projects, save and list scans, build LLM-based project understandings, generate and serve HTML documentation, and accept externally-supplied runtime context + screenshots. The backend orchestrates a ProjectScanner -> ScanStorage -> (optional) LLM provider -> DocumentBuilder -> DocumentStorage flow. A Streamlit front-end (UI/streamlit_app.py) is started separately and interacts with the API. The runtime screenshots show the Streamlit app running on port 8501 and the backend receiving GET and POST requests for project/document endpoints (example project name: AutoDocX).

Runtime flow

1

Start backend API server

Run the FastAPI application under an ASGI server (Uvicorn is used in the observed runtime). The backend exposes endpoints under /api/projects and /health used to scan projects, build LLM understandings, and generate or retrieve HTML documentation.

Evidence: app.main and app.api.project_routes modules present in scan context; terminal screenshot shows backend logs 'Application startup complete.' and multiple GET/POST requests for /api/projects/... endpoints.

2

Start Streamlit UI (optional but present)

Start the Streamlit UI with 'streamlit run UI/streamlit_app.py' which serves at http://localhost:8501. The UI likely calls the backend API endpoints to list projects, request scans, sync documentation, or upload runtime screenshots/context.

Evidence: Screenshot explicitly shows 'streamlit run UI/streamlit_app.py' and 'Local URL: http://localhost:8501'. The project has a UI folder and Streamlit command in screenshot.

3

Trigger a project scan or sync (API call)

A client (Streamlit UI or CLI) calls endpoints such as POST /api/projects/documentation/sync or POST /api/projects/scan. The backend uses ProjectScanner to scan the target local project path and saves the result via ScanStorage.

Evidence: project_routes contains endpoints /api/projects/scan and /api/projects/documentation/sync with called functions project_scanner.scan_project(project_path=...) and scan_storage.save_scan(...). Screenshot shows POST /api/projects/documentation/sync 200 OK in backend logs.

4

LLM-based understanding generation (optional step)

When asked to 'understand' a project, the backend retrieves a stored scan, fetches configured provider/model/api key, and calls ProjectUnderstandingService to generate a structured understanding using the chosen LLM provider. The result is persisted via UnderstandingStorage.

Evidence: Endpoint POST /api/projects/{project_name}/understand in project_routes calls scan_storage.get_scan(...), get_llm_provider(...), get_llm_api_key(...), project_understanding_service.generate_understanding(...), and understanding_storage.save_understanding(...).

5

Document generation and storage

The DocumentBuilder builds HTML documentation from a stored understanding; document_storage.save_document persists the HTML and document metadata. Clients can fetch generated HTML via GET /api/projects/{project_name}/documents/{document_id}/html.

Evidence: project_routes includes generate_project_document which calls document_builder.build_html(...) and document_storage.save_document(...). There is also an endpoint that returns FileResponse for the HTML document.

6

Sync with external runtime context and screenshots

A special endpoint accepts form fields plus uploaded screenshots (multipart form) to attach human-written runtime/context blocks and images to a generated document or sync flow: POST /api/projects/documentation/sync-with-context. Runtime assets are recovered and carried forward on document updates.

Evidence: project_routes includes sync_project_documentation_with_context with Form and File parameters and _recover_runtime_assets_for_document and _get_runtime_context_from_document calls in update_project_document.

Operational notes

  • Backend configuration: the code references get_llm_provider, get_llm_api_key and get_llm_model in app.core.config — ensure environment variables or config file supplying provider selection and API keys are set before calling endpoints that use LLMs.
  • Document and scan storage: app.services.document_storage imports DOCUMENTS_PATH from app.core.config; ensure that DOCUMENTS_PATH exists and is writeable by the process or change configuration to the desired storage location.
  • How to run Streamlit UI: start with 'streamlit run UI/streamlit_app.py' (shown in screenshot). UI will be available at http://localhost:8501 by default.
  • How to run backend: the project uses a FastAPI app (app.main). Start it with an ASGI server (e.g., 'uvicorn app.main:app --reload --port 8000') — the exact run command isn't shown in code scan but Uvicorn-like logs are present. Check repository README or scripts for exact commands.
  • Endpoints of interest: /api/projects/scan, /api/projects/documentation/sync, /api/projects/documentation/sync-with-context, /api/projects/{project_name}/documents and the /understand endpoints. Use them to orchestrate scans, LLM understanding, and HTML generation.
  • Uploads and context: sync-with-context accepts multipart form fields and screenshot files (UploadFile) so the UI or curl/HTTP clients can attach screenshots to documentation.
  • No obvious background queue worker is visible in provided context. If long-running LLM jobs exist, they may run synchronously (blocking request) unless an external worker is implemented elsewhere in the repo; investigate for Celery/RQ/Temporal modules if asynchronous processing is desired.

Runtime risks and gaps

high

LLM API key and provider configuration missing or unclear

The code expects provider names and API keys via get_llm_provider/get_llm_api_key/get_llm_model in app.core.config, but no explicit .env or config sample is visible. Without keys the /understand endpoint will fail or raise errors.

medium

Storage backend persistence details unclear

DocumentStorage and ScanStorage are used, and DOCUMENTS_PATH is referenced, but it's not clear whether storage is filesystem-only or supports S3/db backends. Also there is no explicit migration/initialization step shown for storage directories.

high

No visible authentication/authorization

All listed endpoints appear to be unauthenticated based on provided route definitions. If this service is exposed beyond localhost, it could allow unauthorized scanning, LLM usage (cost), or document retrieval.

medium

Potential blocking/long-running LLM calls

ProjectUnderstandingService likely calls external LLM APIs synchronously. Without an async worker or timeout handling that may block API threads and lead to poor availability under concurrent load.

low

Missing explicit run instructions for backend

While Uvicorn-like logs are present, the exact command used to start the backend is not visible in the provided context. New developers need a clear start script or docker-compose to reproduce environment reliably.

medium

Screenshot upload and size handling not detailed

sync-with-context accepts uploaded screenshots but there is no visible policy for allowed file sizes/types, nor any mention of virus scanning or content moderation. Large uploads may impact storage and performance.

2

Scan a local project

Client calls POST /api/projects/scan with a ProjectScanRequest. ProjectScanner validates the path, enumerates files, parses Python files via PythonCodeParser, analyzes project structure with ProjectAnalyzer, and returns a ProjectScanResponse. ScanStorage persists the stored scan under the configured workspace path.

3

Generate structured project understanding (LLM)

Client calls POST /api/projects/{project_name}/understand. The endpoint loads a stored scan, builds an LLM-ready project context using ProjectContextBuilder, selects and constructs an LLM provider via LLMProviderFactory (OpenAIProvider or AnthropicProvider), sends prompts using provider.generate_json, parses the JSON response, merges deterministic parser/analyzer reference data, validates required fields, and saves the result with UnderstandingStorage.

4

Build HTML documentation

Client calls POST /api/projects/{project_name}/documents to generate HTML. The handler retrieves a stored understanding, DocumentBuilder.build_html renders a professional standalone HTML page (with navigation, API reference cards, signatures, etc.), and DocumentStorage.save_document writes the HTML and JSON metadata to the documents workspace. DocumentStorage.prepare_runtime_assets handles uploaded screenshots when provided.

app.api.project_routesapp.services.document_builderapp.services.document_storageapp.services.understanding_storage
5

Sync complete documentation (all-in-one)

POST /api/projects/documentation/sync triggers the full pipeline: scan_project -> save scan -> locate latest document -> create initial documentation or compare scans and create new version when changes exist -> return current document metadata. This endpoint centralizes the scan->understand->document workflow.

6

Sync with runtime context and screenshots

POST /api/projects/documentation/sync-with-context accepts form fields and uploaded screenshots. Uploaded images are staged and handed to DocumentStorage.prepare_runtime_assets; a RuntimeContextAnalyzer can call a vision-capable LLM provider to analyze screenshots with code context. Returned runtime insights are merged into the final documentation.

Compare scans and list artifacts

The API supports comparing saved scans (POST /api/projects/{project_name}/compare) using ScanComparator.compare_scans, listing scans (/scans) and documents (/documents), and retrieving specific stored scan/understanding/document files and their metadata.

Troubleshooting and Production Notes

high

No authentication or access controls on API endpoints

The API surface (project_routes) exposes scan, LLM calls, and document download endpoints without documented authentication or authorization controls in the scanned code. This may expose project source and LLM keys if deployed publicly.

high

Secrets and credential management

LLM API keys and provider selection are read from environment variables via app.core.config but there is no documented secret management, rotation, or vault integration in the scanned code.

medium

Observability and monitoring

There are no explicit logging, metrics, or tracing integrations visible in the codebase scan that would support production observability, error monitoring, or performance metrics for long-running LLM calls.

medium

Upload and file size handling

While ProjectScanner sets ignored extensions and size limits for scanning, the API paths accepting uploaded screenshots do not show explicit rate limiting or comprehensive validation beyond allowed extensions; large uploads and resource exhaustion are possible.

medium

Testing and CI coverage not exposed

The scan does not include test modules or CI configuration; unit/integration tests and repeatable CI/CD scripts are not present in the provided files.

medium

Deployment and runtime documentation missing

There are start/stop scripts and a requirements.txt, but no container manifests, process manager config, or deployment guidance for production (TLS, reverse proxy, scalability, or persistent storage configuration).

low

Potential LLM response validation and retry behavior

LLM provider implementations parse and validate JSON responses and ProjectUnderstandingService references retry helpers, but global retry/backoff strategies for transient failures and timeouts across all LLM and network calls are not documented as configurable.
