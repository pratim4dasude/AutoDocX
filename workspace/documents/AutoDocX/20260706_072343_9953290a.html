Project Summary

Confirmed: AutoDocX is a FastAPI-based project that scans local Python projects, analyzes their code structure, generates an LLM-based structured 'understanding', and builds/saves HTML documentation. The codebase includes modules for scanning (app.services.project_scanner), parsing Python files (app.services.code_parser.PythonCodeParser), analyzing project-level relationships (app.services.project_analyzer.ProjectAnalyzer), comparing scans (app.services.scan_comparator.ScanComparator), building LLM prompts and interacting with LLM providers via a provider factory (app.services.llm.factory.LLMProviderFactory) with concrete providers for OpenAI (app.services.llm.openai_provider.OpenAIProvider) and Anthropic (app.services.llm.anthropic_provider.AnthropicProvider). Storage components persist scans, understandings, and generated HTML documents to a workspace filesystem (app.services.scan_storage.ScanStorage, app.services.understanding_storage.UnderstandingStorage, app.services.document_storage.DocumentStorage). An API layer (app.api.project_routes) exposes endpoints for scanning, understanding, document generation/update/sync, comparisons and retrievals. A Streamlit UI (ui.streamlit_app) provides a frontend to call the API. Configuration and required-path setup are in app.core.config. Data schemas are defined with Pydantic in app.models.project. Reasonable inferences: The intended runtime is local developer usage: a user points the service (or UI) at a project filesystem path and the service writes workspace artifacts under a workspace directory. LLM calls are synchronous (no async methods in providers) and JSON output from LLMs is parsed into structured understandings. The documentation sync endpoint orchestrates scan -> (compare) -> generate understanding -> render HTML -> save and returns identifiers and metadata.

Architecture Overview

Confirmed: The project is organized into clear layers: API (app.api.project_routes) exposing HTTP endpoints; Services implementing core logic (project_scanner, project_analyzer, code_parser, project_understanding_service, scan_comparator, document_builder); LLM provider abstraction (app.services.llm.base and concrete providers and factory); persistent storage implemented as filesystem-backed stores (scan_storage, understanding_storage, document_storage); configuration (app.core.config); and a UI client (ui.streamlit_app) that calls the API. The FastAPI app entrypoint is app.main. The workspace layout and document/understanding file paths are referenced via constants in app.core.config. Reasonable inferences: The runtime interaction is synchronous: API handlers instantiate or call service classes which in turn use provider factory to create provider instances and call generate_json. Scans and understandings are saved as JSON files under workspace paths. Document HTML files are written to a documents path. The Streamlit UI normalizes API urls and presents document history and sync actions by calling the exposed endpoints. LLM provider selection is controlled by provider name and API key obtained via app.core.config helpers and by request parameters.

Execution Flow

1

All-in-one documentation sync (sync_project_documentation)

API handler orchestrates a full documentation workflow: scans the project, saves the scan, checks for an existing latest document, compares scans if a previous scan exists, creates a new understanding using an LLM when needed, builds HTML from the stored understanding, saves the document metadata and file, and returns the result. If no meaningful changes are detected the existing document is returned.

2

Project scan (scan_project)

ProjectScanner lists filesystem files under the provided path, computes project-level stats (counts, sizes, hashes), invokes PythonCodeParser to parse each Python file for imports, top-level functions/classes/methods and routes, and then calls ProjectAnalyzer to build cross-file mappings, internal dependency graphs, route paths and project statistics. The scan result is returned to the API layer.

3

Save and retrieve scans (scan storage)

ScanStorage persists scan JSONs under workspace/scans/<project_name>/<scan_id>.json and provides methods to save, list and retrieve (including latest) scans. API endpoints call ScanStorage to persist and read scan data.

app.services.scan_storageapp.api.project_routes
4

Generate structured understanding (understand_project)

ProjectUnderstandingService uses ProjectContextBuilder to build an LLM-ready context (modes: 'detailed' and 'llm'), constructs prompts (app.services.llm.prompts), selects an LLM provider using LLMProviderFactory, calls provider.generate_json and parses the JSON result into a structured understanding. UnderstandingStorage persists results under workspace/understandings/<project_name>/<understanding_id>.json.

5

Build and store HTML documentation

DocumentBuilder.build_html converts a stored understanding into a standalone HTML string (escaping content). DocumentStorage.save_document writes the HTML to the documents path (DOCUMENTS_PATH) and persists metadata; API endpoints expose listing and retrieval of documents and serve the HTML file via FileResponse.

Compare scans (compare_project_scans)

ScanComparator.compare_scans compares two stored scans and produces a structured diff describing added/removed/modified files, symbols, routes and internal dependencies. The API exposes comparison endpoints and uses this to decide whether a documentation update is required.

Risks and Gaps

medium

Synchronous/blocking LLM and I/O calls

All provider generate_json methods and most service methods are synchronous. Long-running scans or LLM calls may block the FastAPI worker thread/process, affecting throughput and UI responsiveness. There are no async implementations visible in the scan.

high

LLM output robustness and validation

LLM provider contracts require parsing JSON from model outputs. While BaseLLMProvider.parse_json_response exists to handle plain JSON and some wrapped formats, relying on unvalidated LLM output is risky: malformed JSON or missing fields could propagate into stored understandings and document generation. There is no explicit schema validation of LLM outputs shown in the context (though Pydantic models exist for API messages).

high

Secret management and API key exposure

LLM API keys are retrieved via app.core.config.get_llm_api_key. It's unclear how secrets are stored/loaded (dotenv is imported), whether keys are validated, or whether any secret-scoped logging is prevented. Storing API keys in environment or .env files without guidance is a security risk.

medium

Filesystem-only persistence (no database)

Scans, understandings and documents are persisted to workspace filesystem paths. This design is easy to run locally but may present concurrency, consistency and scaling challenges for multi-user or multi-process deployments. Backups, indexing and transactional guarantees are not apparent.

medium

Error handling and HTTP mapping

Project routes include helper _value_error_status to convert storage lookup errors into HTTP 404, but broader error handling strategy is not fully visible. Unexpected exceptions from LLMs, IO, or parsing may return 500s without structured error payloads. The Streamlit UI provides parsing helpers for backend responses, but server-side error semantics could be documented more clearly.

medium

Limited test and operational metadata visibility

No test modules or CI configuration are present in the supplied context. Operational behavior (logging level, retries, rate-limiting for LLMs, timeouts) and metrics are not visible; this may hinder reliability and debugging in production or heavy usage.

medium

Unclear concurrency / multi-user behavior

Storage paths and naming schemes (scan_id, understanding_id, document_id) are present, but locking/atomic write strategies are not described. Concurrent syncs for the same project may race when reading latest document/scan and writing new artifacts.

low

Incomplete documentation for provider configuration and model selection

There are helper functions get_llm_provider and get_llm_model, and a factory to normalize provider names, but the expected configuration values, supported provider name variants, and how to override model vs. default provider model are not fully documented in the context.
