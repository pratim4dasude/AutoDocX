Project Summary

Confirmed findings: AutoDocX is a FastAPI-based service that scans local Python projects, parses Python source files, analyzes project structure and internal dependencies, stores scan results, builds LLM-ready project context, calls configured LLM providers (OpenAI or Anthropic) to generate a structured "project understanding", and builds/stores HTML documentation generated from that understanding. The API surface is implemented in app.api.project_routes and exposed under /api/projects; application entry point is app.main. Reasonable inferences: The primary goal is automated generation and updating of documentation for Python codebases by combining static analysis and LLM-produced structured descriptions. Configuration and LLM credentials are read via app.core.config; results and artifacts are stored under workspace-like directories (scans, understandings, documents). There is a synchronous (non-async) implementation for scanning, LLM calls, and file IO. Some runtime behavior (exact storage file formats, workspace root constants, and specific Pydantic fields) is not visible in the provided context.

Architecture Overview

Confirmed findings: The project is organized into API (app.api.project_routes), core configuration (app.core.config), data models (app.models.project using Pydantic), services for scanning/parsing/analyzing (app.services.project_scanner, app.services.code_parser, app.services.project_analyzer), services for building context and interacting with LLMs (app.services.project_context_builder, app.services.project_understanding_service, app.services.llm.*), comparison and storage services (app.services.scan_comparator, app.services.scan_storage, app.services.understanding_storage, app.services.document_storage), and an HTML document builder (app.services.document_builder). app.main creates the FastAPI app and registers the router. Reasonable inferences: The system is layered: HTTP API handlers orchestrate workflows using service classes; service classes are thin orchestration points around file parsing, analysis, LLM calls, comparison and persistence. LLM provider implementations conform to a BaseLLMProvider interface and are created by LLMProviderFactory. DocumentBuilder turns a stored understanding into an HTML file which DocumentStorage persists. Storage layout is described in docstrings: scans under workspace/scans/<project_name>/<scan_id>.json and understandings under workspace/understandings/<project_name>/<understanding_id>.json. Exact workspace/constants are defined in app.core.config (DOCUMENTS_PATH, UNDERSTANDINGS_PATH) but not fully shown.

A client calls one of the FastAPI endpoints (root, health, or any /api/projects/* route). app.main registers the router from app.api.project_routes (import alias project_router). The router functions validate requests using Pydantic models from app.models.project.

Project scan is performed (POST /api/projects/scan)

API handler scan_project orchestrates a local project scan: ProjectScanner.scan_project enumerates files, calculates file hashes/sizes, parses Python files using PythonCodeParser.parse_file, and runs ProjectAnalyzer.analyze_project to derive module mappings, internal dependencies and endpoint mappings. The handler then persists the scan using ScanStorage.save_scan.

Stored scans are listed or retrieved

4

Compare two scans (POST /api/projects/{project_name}/compare)

compare_project_scans receives a ScanComparisonRequest, calls ScanComparator.compare_scans to compute added/removed/modified files, symbol changes, route changes and internal dependency diffs. Helper functions in project_routes (comparison_contains_changes and _contains_non_empty_change) are used to determine whether changes are meaningful for documentation updates and to translate storage lookup errors into HTTP 404 status codes.

5

Build LLM-ready context (POST /api/projects/{project_name}/context)

build_project_context accepts ProjectContextRequest and uses ProjectContextBuilder.build_context to convert a stored scan into either a 'detailed' context or a compact 'llm' context intended for prompt input. The resulting ProjectContextResponse is returned.

app.api.project_routesapp.services.project_context_builderapp.services.scan_storage
6

Generate project understanding via LLM (POST /api/projects/{project_name}/understand)

understand_project retrieves the stored scan, builds LLM-ready context via ProjectContextBuilder, constructs system/user prompts using build_project_understanding_prompt and PROJECT_UNDERSTANDING_SYSTEM_PROMPT, creates a provider via LLMProviderFactory.create_provider (matching provider name to OpenAIProvider or AnthropicProvider), and calls provider.generate_json to obtain structured JSON understanding. The result is saved through UnderstandingStorage.save_understanding and returned in ProjectUnderstandingResponse.

7

Generate and store HTML documentation (POST /api/projects/{project_name}/documents)

generate_project_document accepts DocumentGenerationRequest and uses DocumentBuilder.build_html to construct an HTML document from a stored understanding. DocumentStorage.save_document writes the HTML file and associated metadata to DOCUMENTS_PATH and returns metadata (DocumentGenerationResponse). Clients can list documents and download the HTML using list_project_documents and get_project_document_html.

8

Update documentation (POST /api/projects/{project_name}/documents/update)

update_project_document compares a new scan to a previous one via ScanComparator, determines whether documentation-relevant changes exist (using comparison_contains_changes), generates an updated understanding or document as needed, and saves the new document as an update (update_type argument in DocumentStorage.save_document). The API returns DocumentUpdateResponse with update metadata and comparison_summary when available.
