Project Summary

Confirmed findings: - AutoDocX is a Python FastAPI application that scans local Python projects, builds structured project context, asks an LLM to generate a structured project understanding, and converts that understanding into standalone HTML documentation. The application exposes a REST API (app.api.project_routes) and includes a Streamlit UI (ui.streamlit_app) that calls the API. - The project uses storage components to persist scans (ScanStorage), understandings (UnderstandingStorage), and generated HTML documents (DocumentStorage). Scans and understandings are saved as JSON under workspace paths described in code comments (workspace/scans/<project_name>/<scan_id>.json and workspace/understandings/<project_name>/<understanding_id>.json). Document files are stored under a documents path exposed by app.core.config constants. Reasonable inferences (uncertain): - The primary runtime use-case is: scan a project, generate an LLM understanding, and produce HTML documentation; endpoints support individual steps (scan, build context, understand, generate document) and a combined endpoint (/api/projects/documentation/sync) that orchestrates the full workflow. The LLM integration supports multiple providers via a factory (OpenAIProvider, AnthropicProvider) and uses provider, API key, and optional model from app.core.config or request parameters.

Architecture Overview

Confirmed findings: - Components: - API layer: app.api.project_routes defines HTTP endpoints and orchestrates service calls. - Core config: app.core.config supplies runtime configuration (LLM provider selection, API keys, model names) and creates necessary directories. - Models: app.models.project defines Pydantic request/response models used across endpoints. - Scanner & parser: app.services.project_scanner uses app.services.code_parser.PythonCodeParser and app.services.project_analyzer.ProjectAnalyzer to parse files and derive project-wide statistics and relationships. - Context & understanding: app.services.project_context_builder builds LLM-ready context; app.services.project_understanding_service uses app.services.llm.factory.LLMProviderFactory + prompts to call an LLM and produce structured JSON understanding. - LLM providers: app.services.llm.openai_provider.OpenAIProvider and app.services.llm.anthropic_provider.AnthropicProvider implement BaseLLMProvider. - Comparison: app.services.scan_comparator.ScanComparator compares two stored scans to detect added/removed/modified files, symbols, routes, and dependencies. - Storage: app.services.scan_storage.ScanStorage, app.services.understanding_storage.UnderstandingStorage, app.services.document_storage.DocumentStorage persist JSON results and HTML files. - Document builder: app.services.document_builder.DocumentBuilder converts stored understanding into HTML. - UI: ui.streamlit_app provides helper functions to call and display API results. - The API router (router) in project_routes is mounted into the FastAPI app in app.main. - Internal dependencies are explicit: project_routes imports and composes ScanStorage, UnderstandingStorage, ProjectScanner, ProjectUnderstandingService, ScanComparator, DocumentBuilder, DocumentStorage, ProjectContextBuilder, etc. Reasonable inferences (uncertain): - The system is synchronous (no async functions reported) and likely intended for single-request orchestration where LLM calls and file IO are performed synchronously.
